# 面向保险文档的Agentic RAG系统完整落地框架

## 一、核心设计思路与优势

本框架以**多智能体协同为核心**，将保险专业知识、监管规则深度嵌入RAG全链路，通过自主任务规划、多步推理、合规校验三位一体，实现保险文档数据的100%精准提取与合规输出。核心优势在于：合规校验前置、保险知识图谱增强检索、多智能体分工协作、全链路可溯源，彻底解决普通RAG在保险场景中的漏提、错提、无合规等痛点。

---

## 二、整体架构总览

本框架采用**分层多智能体协同架构**，数据流如下：

```
用户查询 → 对话交互模块 → 多智能体调度引擎
                                    ↓
    ┌───────────────────────────────┼───────────────────────────────┐
    ↓                               ↓                               ↓
意图拆解Agent              任务规划Agent               工具调用Agent
    ↓                               ↓                               ↓
    └───────────────────────────────┼───────────────────────────────┘
                                    ↓
                    ┌───────────────┴───────────────┐
                    ↓                               ↓
            保险专属混合检索引擎          文档版本管理模块
                    ↓                               ↓
    ┌───────────────┴───────────────┐       会话记忆模块
    ↓                               ↓
知识图谱检索              语义/关键词检索
    ↓                               ↓
    └───────────────┬───────────────┘
                    ↓
            保险数据精准提取模块
                    ↓
            全链路合规校验模块
                    ↓
            结果格式化输出模块
                    ↓
                用户
```

**协同逻辑说明**：
1. 用户查询进入后，多智能体调度引擎首先调用意图拆解Agent分析查询类型
2. 任务规划Agent根据意图制定执行路径，确定需要调用的工具与检索策略
3. 工具调用Agent执行检索、数据提取等操作
4. 每一步操作结果都经过合规校验模块验证
5. 最终结果由结果输出模块格式化后返回给用户

## 三、核心模块详细设计

### 模块1：保险文档预处理与保险知识图谱构建模块

#### 模块定位
保险文档接入系统的第一道关卡，负责将各类格式的保险文档转换为结构化知识单元，并构建保险领域专属知识图谱。

#### 核心功能
- 多格式文档解析（PDF/Word/Excel/OCR结果）
- 保险条款层级结构解析
- 表格数据精准还原与结构化
- 保险术语标准化映射
- 保险知识图谱构建（实体-关系-属性）
- 文档分块与元数据标注

#### 输入输出标准
| 维度 | 说明 |
|------|------|
| **输入** | 原始保险文档（PDF/Word/Excel/图片） + 文档元数据（版本号、生效日期、文档类型、所属产品） |
| **输出** | 1. 结构化文档分块（带位置标注）；2. 保险知识图谱三元组；3. 术语映射表；4. 文档元数据索引 |

#### 保险场景专属实现逻辑
1. **条款层级解析**：基于保险条款的标准结构（总则、保险责任、责任免除、保险金额、保险期间等）进行规则+LLM混合解析
2. **表格结构化**：针对保险费率表、健康告知表、核保规则表等专用表格，开发专属解析器，保留行列关系与条件逻辑
3. **术语标准化**：构建保险术语库（涵盖人身险/财产险/监管术语），将文档中的同义术语统一映射
4. **知识图谱构建**：定义保险领域专属实体类型（保险产品、责任、免责、等待期、保额等）与关系类型（包含、适用、限制、关联等）

#### 关键技术选型建议
| 组件 | 技术选型 |
|------|----------|
| 文档解析 | PyMuPDF + python-docx + openpyxl |
| OCR | PaddleOCR（开源）或阿里云OCR（商用） |
| 条款解析 | 规则引擎 + LLM微调 |
| 知识图谱 | Neo4j（图数据库） |

### 模块2：多智能体协同调度核心引擎

#### 模块定位
整个框架的"大脑"，负责接收用户查询、拆解意图、规划任务路径、调度子智能体、协调工具调用、处理异常与自修正。

#### 核心功能
- 用户查询意图分类与拆解
- 自主任务路径规划
- 子智能体调度与协调
- 工具调用管理
- 执行状态监控与异常处理
- 错误自修正机制

#### 子智能体设计

| 智能体名称 | 角色定位 | 核心职责 | 工具权限 |
|-----------|---------|---------|---------|
| **意图拆解Agent** | 查询分析师 | 分析用户查询意图，识别查询类型（数据提取/条款咨询/核保理赔/合规咨询等），提取关键实体与约束条件 | 术语映射工具、历史对话查询工具 |
| **任务规划Agent** | 任务调度师 | 根据意图制定详细执行计划，确定需要检索的文档范围、需要调用的工具、数据验证策略 | 文档索引查询工具、知识图谱查询工具 |
| **数据提取Agent** | 保险数据专家 | 负责从检索结果中精准提取保险核心数据，处理嵌套逻辑、条件分支 | 数据提取工具、表格解析工具、逻辑推理工具 |
| **合规校验Agent** | 保险合规官 | 对提取的数据进行合规性验证，确保符合监管要求，检查数据来源准确性 | 监管规则库查询工具、原文比对工具 |
| **结果整合Agent** | 信息整合师 | 整合多源数据，生成结构化输出，标注数据来源，处理跨文档信息冲突 | 数据整合工具、冲突解决工具 |
| **自修正Agent** | 质量监控官 | 监控执行全流程，识别错误与异常，触发重跑或修正流程 | 流程回滚工具、重试工具 |

#### 输入输出标准
| 维度 | 说明 |
|------|------|
| **输入** | 用户查询 + 会话上下文 + 文档索引 |
| **输出** | 结构化任务执行结果 + 执行日志 + 溯源信息 |

#### 保险场景专属实现逻辑
1. **意图分类体系**：预定义保险专属意图分类（如：保额查询、等待期查询、免责条款查询、核保条件查询、理赔材料查询等）
2. **任务模板库**：针对不同意图类型预定义任务执行模板
3. **保险工具集**：构建保险专属工具（条款对比工具、费率计算工具、核保规则匹配工具等）

#### 关键技术选型建议
| 组件 | 技术选型 |
|------|----------|
| 智能体框架 | LangGraph（开源）或 Dify（商用） |
| 任务规划 | LLM（GPT-4o / Claude 3 / Qwen-Max） |
| 工具调用 | OpenAI Function Calling / 通用工具调用协议 |

### 模块3：保险专属混合检索引擎

#### 模块定位
负责根据任务需求，从海量保险文档中精准召回相关内容，支持多维度、多策略的混合检索。

#### 核心功能
- 语义检索
- 关键词检索
- 知识图谱检索
- 元数据过滤
- 检索结果重排序
- 跨文档关联检索

#### 输入输出标准
| 维度 | 说明 |
|------|------|
| **输入** | 查询向量/关键词 + 过滤条件（文档类型、版本、生效日期等） |
| **输出** | 排序后的相关文档分块列表（带相似度评分、位置标注） |

#### 保险场景专属实现逻辑
1. **分层索引构建**：
   - 粗粒度：文档级索引（基于元数据）
   - 中粒度：章节级索引（基于条款结构）
   - 细粒度：段落/句子级索引（语义向量）
2. **保险专用重排序**：基于保险场景特征设计重排序策略（如：条款正文优先级高于注释、最新版本优先级高于旧版本、监管文件优先级高于产品文档）
3. **跨文档关联检索**：基于知识图谱的实体关系，实现跨文档关联召回（如查询某产品条款时，自动召回对应的核保规则、理赔手册）

#### 关键技术选型建议
| 组件 | 技术选型 |
|------|----------|
| 向量数据库 | Milvus / Qdrant（开源）或 Pinecone（商用） |
| 嵌入模型 | BGE-large-zh / Qwen-Embedding（开源）或 OpenAI Embedding（商用） |
| 关键词检索 | Elasticsearch |
| 重排序 | BGE-Reranker-zh（开源） |

### 模块4：保险数据精准提取与逻辑推理模块

#### 模块定位
从检索结果中精准提取保险核心数据，并处理条款中的嵌套逻辑、条件分支，支持多条件组合推理。

#### 核心功能
- 保险核心字段精准提取
- 嵌套逻辑解析
- 条件分支推理
- 表格数据提取
- 跨文档数据关联
- 数据一致性校验

#### 输入输出标准
| 维度 | 说明 |
|------|------|
| **输入** | 检索结果文档分块 + 提取需求 |
| **输出** | 结构化提取结果（JSON格式）+ 来源标注 |

#### 保险场景专属实现逻辑
1. **保险字段Schema**：预定义保险核心字段Schema（保额、免赔额、赔付比例、等待期、犹豫期、保障期限、缴费期限、免责事项、核保条件、健康告知要求、理赔申请材料、理赔时效等）
2. **条件逻辑解析**：针对"如...则..."、"除外..."、"仅限..."等保险条款中常见的条件表述，开发专用解析器
3. **表格数据映射**：将保险费率表、健康告知表等表格数据映射为结构化JSON
4. **多文档融合**：当同一信息在多个文档中出现时，基于版本优先级、文档类型优先级进行融合

#### 关键技术选型建议
| 组件 | 技术选型 |
|------|----------|
| 结构化提取 | LLM Function Calling + 规则校验 |
| 逻辑推理 | LLM（GPT-4o / Claude 3） |
| 数据验证 | Pydantic Schema |

### 模块5：全链路合规与事实性校验模块

#### 模块定位
嵌入框架全链路的合规校验引擎，确保所有数据提取与输出严格符合监管要求，100%匹配原文。

#### 核心功能
- 原文一致性校验
- 监管规则合规性校验
- 数据来源溯源验证
- 误导性表述检测
- 逻辑一致性校验
- 版本有效性校验

#### 输入输出标准
| 维度 | 说明 |
|------|------|
| **输入** | 提取结果 + 原文片段 + 监管规则库 |
| **输出** | 校验通过/不通过 + 校验报告 + 修改建议 |

#### 保险场景专属实现逻辑
1. **校验节点**：在以下关键节点嵌入校验：
   - 数据提取后（原文一致性校验）
   - 结果整合后（逻辑一致性校验）
   - 输出前（监管合规校验）
2. **监管规则库**：构建银保监会监管规则库（如《人身保险产品定名规则》、《健康保险管理办法》等）
3. **原文比对**：采用文本相似度+关键信息点双重比对，确保提取数据100%匹配原文
4. **来源标注**：强制要求所有输出数据标注来源（文档名称、章节、页码）

#### 关键技术选型建议
| 组件 | 技术选型 |
|------|----------|
| 文本比对 | difflib + 语义相似度 |
| 规则引擎 | Drools（开源）或 业务规则管理系统（商用） |
| 合规检测 | LLM微调+提示词工程 |

## 四、全流程业务处理示例

### 场景描述
用户查询：**"我想了解平安e生保2024版的一般医疗保险金的保额是多少？有没有免赔额？如果是质子重离子治疗，赔付比例是多少？需要注意哪些免责事项？"**

### 处理流程

#### 步骤1：意图拆解Agent分析
- **查询类型**：多条件组合数据提取
- **关键实体**：平安e生保2024版、一般医疗保险金、保额、免赔额、质子重离子治疗、赔付比例、免责事项
- **查询意图**：提取特定产品特定责任的多个核心字段

#### 步骤2：任务规划Agent制定计划
1. 检索平安e生保2024版的保险条款
2. 提取"一般医疗保险金"的保额
3. 提取"一般医疗保险金"的免赔额
4. 检索质子重离子责任相关内容
5. 提取质子重离子治疗的赔付比例
6. 提取免责事项
7. 跨文档验证（核保规则/理赔手册）
8. 合规校验
9. 结果整合输出

#### 步骤3：工具调用Agent执行
1. 调用混合检索引擎，检索"平安e生保2024版 保险条款"
2. 调用数据提取工具，从条款中提取一般医疗保险金保额：**400万元**
3. 调用数据提取工具，提取免赔额：**年免赔额1万元**
4. 调用知识图谱检索，找到质子重离子责任相关章节
5. 调用数据提取工具，提取赔付比例：**100%赔付（指定医院）**
6. 调用数据提取工具，提取免责事项列表
7. 调用跨文档检索，确认核保规则中无冲突

#### 步骤4：合规校验Agent验证
- ✅ 保额400万元：与原文条款第5条第1款一致
- ✅ 免赔额1万元：与原文条款第5条第2款一致
- ✅ 质子重离子赔付比例100%：与原文条款第6条第3款一致
- ✅ 免责事项：完整提取，无遗漏
- ✅ 来源标注：所有数据均标注文档名称、章节、页码

#### 步骤5：结果整合Agent输出
**平安e生保2024版 - 一般医疗保险金相关信息**

| 项目 | 内容 | 来源 |
|------|------|------|
| 保额 | 400万元 | 保险条款第5条第1款 |
| 年免赔额 | 1万元 | 保险条款第5条第2款 |
| 质子重离子赔付比例 | 100%（限指定医院） | 保险条款第6条第3款 |

**主要免责事项：**
1. 既往症
2. 美容手术、整形手术
3. 遗传性疾病、先天性畸形
4. 酒后驾驶、无证驾驶
5. 战争、军事冲突

（来源：保险条款第8条）

## 五、核心技术选型清单

### 方案A：轻量化开源方案

| 层级 | 组件 | 技术选型 |

## 六、保险行业专属风险防控与合规管理细则

### 6.1 全链路合规卡点

| 序号 | 卡点位置 | 合规检查内容 | 风险应对方案 |
|------|---------|-------------|-------------|
| 1 | 文档接入 | 文档来源验证、版本有效性、涉密检查 | 建立文档准入审批流程，非官方来源文档禁止接入 |
| 2 | 数据提取 | 原文一致性校验、字段完整性 | 采用"LLM提取+规则校验+人工抽检"三重机制 |
| 3 | 结果生成 | 误导性表述检测、合规表述检查 | 构建禁止表述词库，强制使用标准话术 |
| 4 | 结果输出 | 来源标注完整性、数据准确性 | 未标注来源的结果禁止输出，建立溯源机制 |
| 5 | 日志审计 | 操作日志完整性、查询记录可追溯 | 所有操作记录保存5年以上，支持审计查询 |

### 6.2 保险数据质量标准

| 数据类型 | 准确率要求 | 溯源要求 |
|---------|-----------|---------|
| 保额/保费/免赔额 | 100% | 必须标注具体条款位置 |
| 等待期/犹豫期/保障期限 | 100% | 必须标注具体条款位置 |
| 赔付比例/给付条件 | 100% | 必须标注具体条款位置 |
| 免责事项 | 100%完整 | 必须逐条标注来源 |
| 核保/理赔规则 | 100% | 必须标注规则手册章节 |

### 6.3 风险事件应急响应

| 风险等级 | 事件描述 | 响应流程 |
|---------|---------|---------|
| 一级 | 数据提取错误导致理赔纠纷 | 1. 立即暂停系统服务<br>2. 启动回溯调查<br>3. 修复问题<br>4. 全面复检<br>5. 上报监管 |
| 二级 | 监管规则更新未及时同步 | 1. 立即更新规则库<br>2. 重新校验近期输出<br>3. 通知相关用户 |
| 三级 | 个别数据提取偏差 | 1. 修正数据<br>2. 回溯相关查询<br>3. 优化提取策略 |

## 七、子智能体核心系统提示词模板

### 7.1 意图拆解Agent系统提示词

```
你是一名专业的保险查询意图分析师，拥有10年以上保险行业经验。你的任务是分析用户的保险相关查询，精准拆解查询意图。

【核心职责】
1. 识别查询类型（数据提取/条款咨询/核保咨询/理赔咨询/合规咨询/产品对比等）
2. 提取关键实体（产品名称、险种类型、责任类型、人群、时间范围等）
3. 识别约束条件（版本要求、人群限制、场景条件等）
4. 判断查询的复杂度（单轮/多轮/跨文档）

【输出格式】
请以JSON格式输出：
{
  "intent_type": "查询类型",
  "entities": {"产品名称": "", "责任类型": "", ...},
  "constraints": {"版本": "", "生效日期": "", ...},
  "complexity": "simple/medium/complex",
  "clarification_needed": ["需要澄清的问题列表"]
}

【注意事项】
- 严格基于保险专业知识判断，避免误判
- 如查询存在歧义，在clarification_needed中列出需要澄清的问题
- 输出必须为合法JSON格式
```

### 7.2 数据提取Agent系统提示词

```
你是一名资深的保险数据提取专家，拥有10年以上保险条款解析经验。你的任务是从保险文档中精准提取核心数据，确保100%匹配原文。

【核心职责】
1. 根据提取需求，从给定的文档片段中精准提取目标数据
2. 处理嵌套逻辑、条件分支
3. 解析表格数据
4. 标注数据来源位置

【保险核心字段清单】
- 保额、免赔额、赔付比例、等待期、犹豫期
- 保障期限、缴费期限、缴费方式
- 免责事项、除外责任、责任免除
- 核保条件、健康告知要求、职业类别限制
- 理赔申请材料、理赔时效、理赔流程

【提取规则】
1. 绝对精准：必须100%匹配原文，不做任何演绎、扩写、简化
2. 原文引用：提取结果后必须附上原文引用片段
3. 来源标注：标注文档名称、章节、页码
4. 条件完整：如果数据有前提条件，必须完整提取条件

【输出格式】
{
  "extracted_data": {
    "字段名": {
      "value": "提取值",
      "source": "文档名称 - 章节/页码",
      "original_text": "原文片段",
      "conditions": ["前提条件列表"]
    }
  },
  "confidence": 0.0-1.0,
  "notes": "备注说明"
}

【禁止事项】
- 严禁编造任何保险数据
- 严禁简化或概括原文
- 严禁遗漏前提条件
```

### 7.3 合规校验Agent系统提示词

```
你是一名严格的保险合规官，拥有15年以上保险合规管理经验，熟悉中国银保监会所有监管规定。你的任务是对保险数据提取结果进行全链路合规校验。

【校验内容】
1. 原文一致性校验：提取数据是否100%匹配原文
2. 监管合规校验：是否符合《保险法》、《健康保险管理办法》等监管规定
3. 表述合规校验：是否存在误导性表述、夸大表述
4. 来源完整性校验：数据来源是否完整标注
5. 逻辑一致性校验：数据之间是否存在逻辑冲突

【监管规则要点】
- 不得夸大保险责任
- 不得隐瞒免责事项
- 必须明确等待期、免赔额等限制条件
- 必须使用规范的保险术语
- 不得承诺收益（人身险）

【输出格式】
{
  "compliance_check": {
    "original_consistency": {"passed": true/false, "issues": []},
    "regulatory_compliance": {"passed": true/false, "issues": []},
    "expression_compliance": {"passed": true/false, "issues": []},
    "source_completeness": {"passed": true/false, "issues": []},
    "logical_consistency": {"passed": true/false, "issues": []}
  },
  "overall_pass": true/false,
  "suggestions": ["修改建议列表"]
}

【注意事项】
- 严格把关，任何合规问题都不能放过
- 给出具体的修改建议，而非笼统指出
- 如存在合规问题，overall_pass必须为false
```

### 7.4 任务规划Agent系统提示词

```
你是一名专业的保险任务规划师，拥有丰富的保险知识库使用经验。你的任务是根据用户查询意图，制定详细的执行计划。

【核心职责】
1. 分析查询复杂度，确定需要的步骤
2. 确定需要检索的文档范围（产品条款/核保规则/理赔手册/监管文件等）
3. 确定需要调用的工具
4. 规划执行顺序与依赖关系
5. 定义数据验证策略

【保险任务模板库】
- 简单数据提取：检索→提取→校验→输出
- 多条件提取：多轮检索→分别提取→融合→校验→输出
- 跨文档查询：多文档检索→关联提取→一致性校验→输出
- 条款对比：双文档检索→结构化提取→对比分析→输出

【输出格式】
{
  "task_plan": [
    {
      "step": 1,
      "task": "任务描述",

      "input": "输入参数",

    }
  ],
  "estimated_steps": N,
  "validation_strategy": "验证策略描述",
  "fallback_plan": "备选方案"
}

【注意事项】
- 计划要详细、可执行
- 考虑可能的异常情况，制定备选方案
- 合理安排步骤顺序，提高执行效率
```
